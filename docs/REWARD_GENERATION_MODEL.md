# Reward Generation Model Notes

This note records the current reward-generation findings for Run Planner. It
focuses on reward bag pressure: how many rewards vanilla generates, when those
rewards are removed from a store, and where the planner needs explicit
structure to make simulation deterministic.

## Core Direction

Reward generation pressure belongs to the transition from the current room to
the next offered choices. It should not be attached blindly to the reward inside
the current row.

For a normal linear room:

```text
current room exits -> next offered rooms/rewards
```

So a row's selected reward was generally generated by the previous room's offer
context. The route/timeline planning layer should derive generation context from
the ordered route, rather than making reward bundle declarations infer it from
their own room.

Recommended shape:

```lua
row.rewardGenerationContext = {
    offerCount = previousResolvedRoom.exitCount,
    sourceRowIndex = previousRow.rowIndex,
    sourceRoomKey = previousRow.roomKey,
}
```

Reward items can consume that generated context later, but the biome declaration
should keep topology as topology: exits, cage counts, wheel offer count, forced
structure, and so on.

## Vanilla Reward Store Facts

`ChooseRoomReward(...)` chooses from `run.RewardStores[rewardStoreName]`.

Important behavior:

- The chosen store entry is removed from `run.RewardStores[rewardStoreName]`.
- If no eligible entry exists, vanilla appends a fresh copy of the store and
  tries again.
- The store is not reset by throwing away remaining entries; refill appends.
- `previouslyChosenRewards` is passed through same-generation offers so later
  offers in that batch know what earlier sibling offers already chose.

This means reward bag simulation is not just "did the player take this reward."
Unpicked generated offers can still remove entries from the same store before
the player chooses a path.

## Simulation Boundary

The planner should distinguish:

- Known store and known reward: consume that exact reward.
- Known store and unknown reward: consume one eligible non-reserved entry from
  that store.
- Unknown store: not enough information for exact bag simulation.

Future planned concrete rewards should be reserved before consuming unknown
offers. Otherwise an unknown sibling offer can incorrectly consume a reward that
the user planned later.

## Straightforward Biomes

### F, G, P

These can use the previous room's offer topology for normal route rows:

```text
next reward offer count = previous room exitCount
```

The current room declaration should continue to own `exitCount`; the planning
walk derives the reward generation pressure for the next row from that prior
room.

Preboss is different. F/G/P preboss rooms expose both:

- shop offers
- a free RunProgress reward branch

Those are not ordinary next-door exits. They should stay modeled as explicit
preboss reward nodes with their own timing/choice semantics.

### I

Tartarus has a clockwork progression axis. A two-exit room can offer one
clockwork goal/progression exit and one optional reward/extension exit.

For normal reward-bag pressure:

```text
optional reward offer count = max(exitCount - 1, 0)
```

The reserved clockwork goal path should not be treated as a normal RunProgress
or TartarusRewards bag offer.

## Ambiguous Biomes That Need Explicit Structure

### H Fields

Fields is the worst case for bag simulation.

Confirmed vanilla behavior:

- `BaseH` uses `IndividualRewardStore = "RunProgress"` and
  `TargetMetaRewardsRatio = 0.0`.
- H door cage counts are `MinDoorCageRewards = 2` and
  `MaxDoorCageRewards = 3`.
- During exit unlock, every offered H door with `MaxCageRewards` builds
  `doorRoom.CageRewards`.
- Each cage reward calls `ChooseRoomReward(...)`.
- Each cage reward appends to the same `rewardsChosen` batch.

So a 2-reward cage room consumes two RunProgress entries. A 3-reward cage room
consumes three RunProgress entries. The unpicked offered door can therefore
deplete the RunProgress bag before the later planned path matters.

Bounded simulation is not good enough here:

- Low-roll simulation refills too late.
- High-roll simulation refills too early.
- The route can become genuinely ambiguous.

Current modeling direction:

- H reward routing should force deterministic generated structure.
- The user should select the second door structure, not its rewards.
- This is needed only for the ambiguous H choice rows, not as a global model.

Structure enum:

```lua
structure = "CombatCage2" | "CombatCage3" | "Miniboss" | "Bridge"
```

Pressure mapping:

```text
CombatCage2 -> 2 RunProgress offers
CombatCage3 -> 3 RunProgress offers
Miniboss     -> 1 RunProgress offer, Boon-only
Bridge       -> 0 RunProgress offers
```

When the picked side is a combat cage and vanilla shares the same cage count
across offered cage doors, the sibling cage count can match the picked cage
count. When the picked side is Bridge or Miniboss, the sibling structure must
say whether the other generated door was `CombatCage2`, `CombatCage3`,
`Miniboss`, or `Bridge`.

The planner should not ask reward validation to guess H randomness. It should
make H generation deterministic first, then simulate that deterministic plan.

### O Thessaly

Thessaly ship wheels have a smaller version of the same problem.

The ship wheel can generate one or two reward choices. The player picks one,
but if two were generated, the unpicked sibling reward still came from the same
store batch.

Current modeling direction:

```lua
wheelOfferCount = 1 | 2
```

Meaning:

- `1`: only the selected wheel reward was generated.
- `2`: selected reward plus one unpicked sibling reward were generated.

This belongs on the O encounter reward node. It should not be inferred from
generic room exits.

## Design Rules

- Biome declarations own topology and structure.
- Reward declarations own bundles, primitives, shop surfaces, and structural
  filters.
- Route/timeline planning derives generation context from ordered route
  structure.
- Runtime should execute the prepared plan; it should not compensate for vague
  reward bag state.
- H and O should become deterministic at the generation-structure level before
  exact reward bag simulation is trusted.
